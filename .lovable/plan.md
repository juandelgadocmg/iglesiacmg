

## Plan: QR-Based Attendance, Fix Reuniones, Polish Peticiones and Certificados

This plan addresses all the issues you raised: fixing the Reuniones attendance flow, adding QR codes for attendance tracking, enabling Google-based check-in for general services, and ensuring Peticiones and Certificados work correctly end-to-end.

---

### 1. Generate QR Codes for Each Persona

**Database**: Add a `qr_code` text column to `personas` with a default value of `gen_random_uuid()` so every person gets a unique QR identifier automatically (existing personas get one via a backfill).

**Migration**:
- `ALTER TABLE personas ADD COLUMN qr_code text DEFAULT gen_random_uuid()::text;`
- `UPDATE personas SET qr_code = gen_random_uuid()::text WHERE qr_code IS NULL;`
- `ALTER TABLE personas ADD CONSTRAINT personas_qr_code_key UNIQUE (qr_code);`

**Frontend**: 
- Install `qrcode.react` package to render QR codes.
- On the **PersonaPerfilPage**, add a QR code card that renders a QR image from `persona.qr_code`. Include a download/print button.
- When creating a new persona, the `qr_code` is auto-generated by the database default.

---

### 2. Fix Reuniones Attendance Flow

The current flow loads all personas and lets you toggle attendance. The issue is likely that the `ServicioFormDialog` only saves basic fields (nombre, fecha, tipo, hora, lugar, predicador, descripcion) but doesn't include `sede`, `dia_reunion`, `aforo`, or `habilitado_reserva` fields that the card UI displays.

**Fix ServicioFormDialog**: Add missing fields (`sede`, `dia_reunion`, `aforo`, `habilitado_reserva`) to the form and payload.

**Fix attendance saving**: The upsert uses `onConflict: "servicio_id,persona_id"` which matches the existing unique constraint -- this should work. Verify and fix any edge cases in the `ReporteDetalleView`.

---

### 3. QR Scanner for Reuniones Attendance

**New component**: `QrAttendanceScanner` using `html5-qrcode` library.
- Add a "Escanear QR" button in `ReporteDetalleView`.
- When scanned, look up the persona by `qr_code` value.
- Auto-toggle their attendance to present with a success toast showing the person's name.
- Support continuous scanning mode (scan multiple people in sequence).

---

### 4. Google Account Check-in for General Services

For general services (Culto general type), allow members who have a Google account linked to check in themselves.

**Approach**: 
- Create a new public-facing route `/check-in/:servicioId` that shows the service name and a "Check in con Google" button.
- The button uses `lovable.auth.signInWithOAuth("google")` to authenticate.
- After authentication, match the Google email against `personas.email`.
- If a match is found, upsert an attendance record as present.
- If no match, show an error message suggesting they contact the church admin.
- The reunion detail view gets a "Compartir enlace de check-in" button that copies the URL.

---

### 5. Polish Peticiones Module

Current state looks functional. Fixes needed:
- **Panel view**: The filtered list shows count but no actual list/table of petitions -- add a table below the search showing the filtered results.
- **Gestionar view**: The "Cancelar" button on each card calls `onDelete` which is confusing -- rename to "Eliminar" or add confirmation.
- Ensure creating a petition from PersonaFormDialog (during persona registration) correctly populates the `tipo` field and shows up in the Peticiones module.

---

### 6. Polish Certificados Module

Current state looks functional. Fixes needed:
- **PDF generation**: Implement actual PDF download using `jsPDF` instead of placeholder toast. The PDF should include: church name, pastor name (from configuracion_iglesia), certificate type, person name, date, and certificate code.
- Ensure the certificate types match the document: Membresia, Bautismo, Confirmacion, Matrimonio, Presentacion de ninos, Donacion, Carta de transferencia.

---

### 7. Cross-Module Data Synchronization

Ensure query invalidation propagates across modules:
- When a persona is created/updated, invalidate `personas`, `peticiones_oracion`, `certificados`, and `asistencia` queries.
- When a peticion is created from PersonaFormDialog, invalidate `peticiones_oracion` query key.
- When attendance is recorded, invalidate dashboard stats.
